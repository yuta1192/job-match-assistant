# 面談説明スクリプト（5分）

転職面談で Job Match Assistant を説明するための台本。
各機能について「なぜその設計にしたか」を自分の言葉で言えるようにする。

---

## 30秒版（つかみ）

> Rails の実務経験を土台に、TypeScript / React / LLM API / Docker / CI/CD を実装でキャッチアップするために、転職活動を効率化する「Job Match Assistant」という個人開発アプリを作りました。
> 求人本文とスキルシートをもとに、技術スタック抽出・マッチ理由・懸念点・面談で確認すべき点・返信文案を LLM で生成し、応募ステータスと面談メモまで管理できます。

---

## 5分版（構成）

### 1. 背景（30秒）
転職活動で複数求人を比較する際、技術スタックの読み取り、自分の経験とのマッチ整理、返信文作成に時間がかかっていた。これを LLM で効率化しつつ、モダンスタックを実装で学ぶために開発した。

### 2. 技術構成（30秒）
フロントは Next.js / TypeScript、バックは Rails API + PostgreSQL、LLM は OpenAI/Anthropic の公式 API を直接利用。Docker で開発環境を統一し、GitHub Actions で RSpec を自動実行。

### 3. 機能デモ（2分）
- 求人を登録 → 分析実行で技術スタック抽出・マッチ分析
- スキルプロフィールとの突き合わせで「マッチ点 / 足りない点 / 面談確認点」を生成
- 返信文を生成（AI 感を出さず、経験との接点を明確にするトーン）
- 応募ステータス・面談メモを管理

### 4. 設計上の工夫（1.5分）
- **型から仕様を表現** — TypeScript で主要データ型を先に定義し、画面・API がそれを参照
- **LLM 出力の安定化** — JSON 形式で固定、`raw_response` を保存して再現性・デバッグ性を確保
- **責務分離** — LLM 呼び出しは Service Object に切り出し、コントローラを薄く保つ
- **テスト** — LLM 部分は mock し、push 時に CI でテストを走らせる

### 5. AI 活用方針（30秒）
Claude Code / Cursor は実装案比較・テスト観点・リファクタ検討・ドキュメント補助に利用。設計判断・レビュー・動作確認は自分で実施し、生成コードはそのまま採用せず責務・例外・セキュリティを確認して取り込んだ。

---

## 機能ごとの「説明できる」チェックリスト

- **求人登録**: なぜこのデータ構造か / URL と本文をどう扱うか / ステータス管理の設計
- **スキルプロフィール**: 分析時にどの情報を LLM へ渡すか / 個人情報をどこまで扱うか
- **LLM 分析**: prompt 設計 / 出力形式の安定化 / エラー時の挙動 / raw_response 保存理由 / コスト意識
- **返信文生成**: AI 感を減らす指示 / 経験を盛りすぎない制御 / 複数トーンの扱い
- **Docker**: なぜ使うか / 3サービスの接続 / 環境変数の扱い
- **GitHub Actions**: いつ何をテストするか / LLM 部分の mock
